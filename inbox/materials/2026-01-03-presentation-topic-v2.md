# 발표 주제 개발 v2: AI 시대의 개발자, 남는 것은 무엇인가

## 핵심 메시지

> "AI 시대, 기술은 AI한테 배우면 된다. 개발자에게 남는 건 '왜'와 '무엇을'이다."

---

## 발단과 진화

```
v1: DevOps 용어 정리
    ↓
v2: Why/What/How 프레임 발견
    ↓
v3: 커리어란 방향이다
    ↓
v4: AI 시대의 개발자는 뭘 해야 하는가 ← 현재
```

단순 용어 정리에서 시작해서, 결국 **"개발자로서 나는 뭘 해야 하는가"**라는 질문에 도달함.

---

## 배경: 왜 이 얘기를 하게 됐나

### 다방면의 경험에서 같은 결론

```
개발      → 넓고 깊게 해봄
운영      → 넓고 깊게 해봄
생각정리  → 넓고 깊게 해봄
AI 서비스 → 넓고 깊게 해봄
프로세스  → 넓고 깊게 해봄
          ↓
전부 같은 레이어 구조가 있더라
          ↓
DevOps가 가장 설명하기 좋은 예시
```

### 발표에서 언급할 부분

> "사실 DevOps만 그런 게 아니라, 생각정리 방법론도, AI 도구 선택도, 파이프라인 설계도 다 같은 구조더라. 오늘은 가장 익숙한 DevOps로 얘기하지만, 이 프레임은 어디든 적용된다."

---

## 발표 구조

### 1. Hook: 기술 용어 홍수 + 냉소 + 그런데...

```
"DevOps, GitOps, MLOps, AIOps, DevSecOps...
 솔직히 냉소적이었다. '또 새로운 -Ops냐.'
 근데 정리해보니까 패턴이 보이더라.

 그리고 깨달았다.
 기술은 이제 AI한테 물어보면 된다.
 그럼 개발자인 나는 뭘 해야 하지?"
```

**의도:** 청중도 느꼈을 피로감에서 시작 → 공감 → 질문으로 전환

---

### 2. 발견: Why/What/How 프레임

```
추상적 ◀───────────────────────────────▶ 구체적

        Why          What          How
         │            │             │
DevOps   ████████████████           ░░░░░░░░  (How 자유)
GitOps   ░░░░░░░░░░░░████████████████████████  (How 강제)
```

**핵심 인사이트:**

| 구분 | 성격 | 예시 |
|------|------|------|
| Philosophy | Why + 방향 | "검증 불가, 문화" |
| Methodology | How 강제 | "검증 가능, 체크리스트" |

```
"DevOps 한다" → 검증 불가 (문화/철학)
"GitOps 한다" → 검증 가능 (제약 체크리스트 존재)
```

---

### 3. 질문: AI가 다 해주는 시대, 개발자는 뭘 하나?

```
과거:  기술을 아는 것 = 경쟁력
       "이 프레임워크 써봤어요" = 차별점

현재:  기술은 AI한테 물어보면 됨
       "Claude야, React로 이거 만들어줘"

그럼 뭐가 남지?
       ↓
철학, 주관, 그리고 그 근거
```

**이 발표가 기술 발표가 아닌 이유:**

> "이제 기술은 AI로 배우면 된다. 자기 자신의 철학과 주관, 그리고 그 근거가 뚜렷해야 한다. 기술적 도메인에 있어서 말이다."

---

### 4. 답: 커리어는 동기에서 온다

**커리어의 정의:**

```
커리어 = 방향
       = 나를 완성시키러 가는 길
       = 나의 삶의 목표

어디서 오는가?
       = 나의 동기
       = 나의 이키가이
```

**40이 되어서야 깨달은 것들:**

> "왜 위인전기를 읽는지, 왜 다독·다작·다상량을 해야하는지 이제야 깨닫는다."

> "롤모델이 누구였는가? 이런 게 왜 중요한지."

**코드에도 적용:**

```
다독  → 남의 코드를 많이 읽어라
다작  → 직접 많이 짜봐라
다상량 → 왜 이렇게 짰는지 생각해라
```

> "무조건 만들기만 하신다면 제발 남의 코드를 읽고 이해해보라."

> "반대로 남의 것을 베끼기만 한다면 내재화해서 직접 만들어봐라."

---

### 5. 에피소드: 주니어에게 배운 것

```
"최근에 주니어 개발자와 사석에서 얘기를 나눴다.
 내가 조언한다고 했는데, 집에 가서 이불킥을 했다.
 돌이켜보니 내가 배운 거였다."
```

**그 대화에서 던진 질문:**

> "나는 네가 뭘 하고 싶은지가 궁금해. 궁극적으로 뭘 만들고 싶니?"

**왜 이 질문이 중요한가:**

> "만들고 싶은 것에는 정말 밀도 높은 함축된 그 사람의 방향이 보인다."

---

### 6. 선언: Why/What으로 소통한다

```
앞으로 나는:
├── Why/What을 공유한다
├── How는 강요하지 않는다
└── 방법론(PARA, Zettelkasten, GitOps...)은 각자 선택
```

**청중에게:**

> "그저 남이 이렇게 하라는 단편적인 말로 당신의 커리어를 정하지 않았으면 좋겠다."

**방법론 소개 (How의 예시들):**

- 생각정리: PARA, Zettelkasten, ...
- 운영: GitOps, ChatOps, ...
- 개발: TDD, DDD, ...

→ 뭘 쓰든 상관없다. 당신의 Why가 먼저다.
